## v10.6 — Zombie-tunnel leak prevention + interface coverage

### Problem

v10.5 could open the internet when the WireGuard tunnel service reported **RUNNING** but traffic was not actually flowing (zombie tunnel), or when only one of two connectivity checks would have failed. Tethering/PPP interfaces were not covered by block rules.

### Fixes

| Area | Change |
|------|--------|
| **Safe open** | `Test-SafeToOpen` requires tunnel RUNNING **and** dual-host `Test-Internet` (1.1.1.1 + 1.0.0.1) before `Disable-Block` |
| **Zombie handling** | Main loop treats tunnel-up/no-internet as unhealthy; triggers block + recovery |
| **3min wait** | Success requires `Test-SafeToOpen`, not tunnel status alone |
| **Firewall** | `KS-Block-RemoteAccess-Out` and `KS-Block-PPP-Out` for tethering/PPP leaks |
| **WARP IPs** | Monitor refreshes Cloudflare server CIDRs every ~5 minutes via `Get-ResolvedServerIP` |
| **Log mutex** | All components skip log write if mutex times out (no corruption) |
| **pwsh** | Monitor detection + WMI subscription include `pwsh.exe` |
| **Mutex errors** | Monitor logs fatal mutex errors instead of silent `exit 0` |

### Upgrade

```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
.\install.ps1
```

Re-run with the same parameters if you used custom mode (`-CustomConfig`, etc.).
