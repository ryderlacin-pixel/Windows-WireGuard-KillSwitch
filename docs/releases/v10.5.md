## v10.5 — AbandonedMutexException fix (monitor respawn)

### Critical fix

v10.4 added `Global\WGMainMonitorMutex` with:

```powershell
catch { exit 0 }  # caught AbandonedMutexException too
```

When the monitor is killed (Task Manager, crash), the mutex is **abandoned**. Per Microsoft docs, the next `WaitOne` acquires the mutex but may throw `AbandonedMutexException`. The blanket `catch { exit 0 }` treated that as "duplicate instance" and exited — WMI/repair could not restore the monitor.

**v10.5** adds `Wait-NamedMutex` everywhere mutexes are used:

```powershell
function Wait-NamedMutex([System.Threading.Mutex]$Mutex, [int]$TimeoutMs) {
    try { return $Mutex.WaitOne($TimeoutMs) }
    catch [System.Threading.AbandonedMutexException] { return $true }
}
```

- Main monitor: abandoned → continue running (respawn succeeds)
- Live duplicate: `WaitOne(0)` returns `$false` → `exit 0` (unchanged)
- Tunnel reinstall + log mutexes: same helper (defensive)

### Install / upgrade
```powershell
Set-ExecutionPolicy Bypass -Scope Process -Force
.\install.ps1
```
